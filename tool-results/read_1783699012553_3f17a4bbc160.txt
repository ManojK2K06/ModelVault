   165в†’  1164в†’  1164в†’
   166в†’  1165в†’  1165в†’6.1 Feature 1: Model Signing and Verification
   167в†’  1166в†’  1166в†’
   168в†’  1167в†’  1167в†’6.1.1 How Model Signing Works
   169в†’  1168в†’  1168в†’
   170в†’  1169в†’  1169в†’Model signing implements a Sigstore-inspired keyless signing workflow.
   171в†’  1170в†’  1170в†’When a user clicks 'Sign Model' on a model version, the following
   172в†’  1171в†’  1171в†’process executes: The signing-service receives a sign request containing
   173в†’  1172в†’  1172в†’the model_version_id and the user's JWT access token. The service first
   174в†’  1173в†’  1173в†’retrieves the model's SHA-256 and Merkle root hash from the
   175в†’  1174в†’  1174в†’model_versions table. It then constructs a signing payload that includes
   176в†’  1175в†’  1175в†’the model ID, version string, SHA-256 hash, Merkle root, file size,
   177в†’  1176в†’  1176в†’format, and the current timestamp. This payload is serialized to JSON
   178в†’  1177в†’  1177в†’and canonicalized (sorted keys, no whitespace) to ensure deterministic
   179в†’  1178в†’  1178в†’hashing. The payload is then hashed with SHA-256 to produce the digest
   180в†’  1179в†’  1179в†’that will be signed.
   181в†’  1180в†’  1180в†’
   182в†’  1181в†’  1181в†’The signing-service generates a short-lived ephemeral key pair (ECDSA
   183в†’  1182в†’  1182в†’P-256) for this signing operation. It then contacts the Fulcio
   184в†’  1183в†’  1183в†’certificate authority (or an internal Fulcio-compatible CA for
   185в†’  1184в†’  1184в†’self-hosted deployments) with an OIDC identity token (obtained from the
   186в†’  1185в†’  1185в†’user's session) and the ephemeral public key. Fulcio issues a
   187в†’  1186в†’  1186в†’short-lived X.509 certificate that binds the ephemeral public key to the
   188в†’  1187в†’  1187в†’user's OIDC identity (email). The signing-service signs the payload
   189в†’  1188в†’  1188в†’digest with the ephemeral private key, producing an ECDSA-SHA256
   190в†’  1189в†’  1189в†’signature. The signature, the X.509 certificate chain, and the signing
   191в†’  1190в†’  1190в†’payload are all stored in the signatures table. The ephemeral private
   192в†’  1191в†’  1191в†’key is then securely erased from memory. The certificate chain is also
   193в†’  1192в†’  1192в†’submitted to a Rekor transparency log (or an internal immutable log for
   194в†’  1193в†’  1193в†’self-hosted deployments), which returns a signed inclusion proof.
   195в†’  1194в†’  1194в†’
   196в†’  1195в†’  1195в†’Verification reverses this process: given a model file and a signature
   197в†’  1196в†’  1196в†’record, the system recomputes the SHA-256 hash of the file, reconstructs
   198в†’  1197в†’  1197в†’the signing payload, verifies the ECDSA signature against the X.509
   199в†’  1198в†’  1198в†’certificate from the stored certificate chain, validates that the
   200в†’  1199в†’  1199в†’certificate was issued by a trusted CA, checks that the certificate has
   201в†’  1200в†’  1200в†’not expired, and optionally verifies the Rekor inclusion proof. If all
   202в†’  1201в†’  1201в†’checks pass, the verification returns 'VERIFIED' with the signer's
   203в†’  1202в†’  1202в†’identity. If any check fails, it returns 'FAILED' with a specific reason
   204в†’  1203в†’  1203в†’(hash mismatch, signature invalid, certificate expired, untrusted CA,
   205в†’  1204в†’  1204в†’etc.).
   206в†’  1205в†’  1205в†’
   207в†’  1206в†’  1206в†’6.1.2 Merkle Tree Integrity
   208в†’  1207в†’  1207в†’
   209в†’  1208в†’  1208в†’During upload, the model file is split into 4MB chunks. Each chunk is
   210в†’  1209в†’  1209в†’hashed with SHA-256, producing a list of leaf hashes. These leaf hashes
   211в†’  1210в†’  1210в†’are arranged into a binary Merkle tree: pairs of leaf hashes are
   212в†’  1211в†’  1211в†’concatenated and hashed to produce parent nodes, up to a single root
   213в†’  1212в†’  1212в†’hash. The Merkle root is stored in the model_versions table alongside
   214в†’  1213в†’  1213в†’the flat SHA-256 of the complete file. During verification, both the
   215в†’  1214в†’  1214в†’flat hash and the Merkle root are verified. The Merkle tree enables
   216в†’  1215в†’  1215в†’efficient proof that a specific chunk of the model file has not been
   217в†’  1216в†’  1216в†’tampered with, without requiring the entire file to be re-downloaded.
   218в†’  1217в†’  1217в†’This is particularly useful for partial model updates and for verifying
   219в†’  1218в†’  1218в†’models in distributed storage systems where different chunks may reside
   220в†’  1219в†’  1219в†’on different nodes.
   221в†’  1220в†’  1220в†’
   222в†’  1221в†’  1221в†’6.2 Feature 2: AI-SBOM Generation
   223в†’  1222в†’  1222в†’
   224в†’  1223в†’  1223в†’6.2.1 How SBOM Generation Works
   225в†’  1224в†’  1224в†’
   226в†’  1225в†’  1225в†’AI-SBOM generation is a fully deterministic, non-AI process that uses
   227в†’  1226в†’  1226в†’static analysis to extract metadata from model artifact files. When a
   228в†’  1227в†’  1227в†’model is uploaded (or when the user explicitly triggers SBOM
   229в†’  1228в†’  1228в†’generation), the sbom-service retrieves the model file from MinIO and
   230в†’  1229в†’  1229в†’analyzes it based on its detected format. The analysis extracts: model
   231в†’  1230в†’  1230в†’architecture metadata (number of layers, parameter count, tensor
   232в†’  1231в†’  1231в†’dimensions, dtype), the complete dependency tree of Python packages
   233в†’  1232в†’  1232в†’required to load and run the model (extracted from any embedded
   234в†’  1233в†’  1233в†’requirements.txt, setup.py, pyproject.toml, or from the model card
   235в†’  1234в†’  1234в†’metadata), a list of all third-party model components (base model
   236в†’  1235в†’  1235в†’references, adapter configurations, tokenizer files), hardware
   237в†’  1236в†’  1236в†’requirements (inferred from tensor sizes and dtype), and training data
   238в†’  1237в†’  1237в†’provenance metadata if available from the model card or embedded
   239в†’  1238в†’  1238в†’metadata.
   240в†’  1239в†’  1239в†’
   241в†’  1240в†’  1240в†’For SafeTensors format files, the parser reads the JSON header at the
   242в†’  1241в†’  1241в†’start of the file, which contains the tensor metadata including names,
   243в†’  1242в†’  1242в†’shapes, and dtypes. It calculates the total parameter count by
   244в†’  1243в†’  1243в†’multiplying the dimensions of each tensor. It identifies the model
   245в†’  1244в†’  1244в†’architecture by matching tensor naming patterns (e.g.,
   246в†’  1245в†’  1245в†’'transformer.h.{N}.attn.c_attn.weight' indicates a GPT-2 style
   247в†’  1246в†’  1246в†’architecture). For GGUF files, the parser reads the GGUF header which
   248в†’  1247в†’  1247в†’contains key-value metadata including the model architecture,
   249в†’  1248в†’  1248в†’quantization level, and embedding dimensions. For ONNX files, the parser
   250в†’  1249в†’  1249в†’uses the ONNX protobuf schema to read the model graph, extracting
   251в†’  1250в†’  1250в†’operation types, input/output shapes, and opset version. For PyTorch
   252в†’  1251в†’  1251в†’.bin or .pt files, the parser loads the file using PyTorch's safe_load
   253в†’  1252в†’  1252в†’function (which uses zip-based inspection without executing any code)
   254в†’  1253в†’  1253в†’and extracts the state dict keys, tensor shapes, and dtypes.
   255в†’  1254в†’  1254в†’
   256в†’  1255в†’  1255в†’After extraction, the sbom-service cross-references all discovered
   257в†’  1256в†’  1256в†’dependencies against the OSV (Open Source Vulnerabilities) database API
   258в†’  1257в†’  1257в†’to identify known vulnerabilities. Each dependency is looked up by its
   259в†’  1258в†’  1258в†’package name and version. If a CVE is found, it is recorded with its
   260в†’  1259в†’  1259в†’severity (CRITICAL, HIGH, MEDIUM, LOW), CVSS score, and a link to the
   261в†’  1260в†’  1260в†’advisory. The final SBOM is output in CycloneDX ML-BOM 1.0 JSON format,
   262в†’  1261в†’  1261в†’which extends the standard CycloneDX format with ML-specific fields
   263в†’  1262в†’  1262в†’including model architecture, parameter count, training data provenance,
   264в†’  1263в†’  1263в†’and hardware requirements. The SBOM JSON is stored in the sboms table
   265в†’  1264в†’  1264в†’and is also available for download via the API endpoint GET
   266в†’  1265в†’  1265в†’/api/v1/models/{model_id}/sbom.
   267в†’  1266в†’  1266в†’
   268в†’  1267в†’  1267в†’6.3 Feature 3: Pre-Deployment Quarantine Sandbox
   269в†’  1268в†’  1268в†’
   270в†’  1269в†’  1269в†’6.3.1 Sandbox Architecture
   271в†’  1270в†’  1270в†’
   272в†’  1271в†’  1271в†’The quarantine sandbox is the most critical security feature of
   273в†’  1272в†’  1272в†’ModelVault. It provides a completely isolated execution environment
   274в†’  1273в†’  1273в†’where a model is loaded and exercised with a battery of behavioral tests
   275в†’  1274в†’  1274в†’to detect malicious behavior before the model is approved for
   276в†’  1275в†’  1275в†’deployment. The sandbox uses Firecracker microVMs, which are lightweight
   277в†’  1276в†’  1276в†’virtual machines that boot in under 125 milliseconds with a dedicated
   278в†’  1277в†’  1277в†’kernel, providing hardware-level isolation from the host system. Each
   279в†’  1278в†’  1278в†’sandbox job gets its own Firecracker VM with a minimal root filesystem
   280в†’  1279в†’  1279в†’(based on Alpine Linux, approximately 5MB), a Python runtime (CPython
   281в†’  1280в†’  1280в†’3.12), a PyTorch/CUDA runtime (only CPU mode for security), and the
   282в†’  1281в†’  1281в†’model file under test.
   283в†’  1282в†’  1282в†’
   284в†’  1283в†’  1283в†’The VM is configured with a strict network policy enforced by iptables:
   285в†’  1284в†’  1284в†’all outbound network connections are blocked by default. The only
   286в†’  1285в†’  1285в†’allowed outbound traffic is a DNS resolution to a mock DNS server that
   287в†’  1286в†’  1286в†’returns NXDOMAIN for all queries, used to test whether the model
   288в†’  1287в†’  1287в†’attempts to perform DNS lookups (which would indicate potential data
   289в†’  1288в†’  1288в†’exfiltration). All file system access is monitored using eBPF programs
   290в†’  1289в†’  1289в†’attached to the VFS layer, which capture every open(), read(), write(),
   291в†’  1290в†’  1290в†’and unlink() syscall with the full path, flags, and return value. The
   292в†’  1291в†’  1291в†’eBPF programs are compiled from BPF C code using the bcc toolkit and
   293в†’  1292в†’  1292в†’loaded into the VM's kernel at boot time. Syscall events are streamed to
   294в†’  1293в†’  1293в†’a sidecar process over a virtio-serial channel, which aggregates them
   295в†’  1294в†’  1294в†’and writes the structured event stream to the sandbox_jobs.result_json
   296в†’  1295в†’  1295в†’field in the database.
   297в†’  1296в†’  1296в†’

   376в†’  1375в†’  1375в†’document is immutable once written and is viewable in the frontend
   377в†’  1376в†’  1376в†’sandbox results page.

   379в†’  1378в†’  1378в†’6.3.3 Sandbox Job Lifecycle
   380в†’  1379в†’  1379в†’
   381в†’  1380в†’  1380в†’A sandbox job goes through the following lifecycle states: QUEUED (job
   382в†’  1381в†’  1381в†’created, waiting for a VM slot), PROVISIONING (Firecracker VM is being
   383в†’  1382в†’  1382в†’created and the model file is being copied into the VM's rootfs),
   384в†’  1383в†’  1383в†’RUNNING (all behavioral tests are executing inside the VM), COMPLETED
   385в†’  1384в†’  1384в†’(all tests finished, results captured, VM terminated), FAILED (VM
   386в†’  1385в†’  1385в†’crashed, timeout, or internal error), CANCELLED (user cancelled the job
   387в†’  1386в†’  1386в†’before completion). The sandbox-service maintains a pool of
   388в†’  1387в†’  1387в†’pre-provisioned VM slots (configurable, default 4 per node) to minimize
   389в†’  1388в†’  1388в†’provisioning latency. When a job is QUEUED, it is assigned to the next
   390в†’  1389в†’  1389в†’available slot. If no slot is available within 30 seconds, the job
   391в†’  1390в†’  1390в†’remains in QUEUED status and the frontend displays a queue position
   392в†’  1391в†’  1391в†’indicator. The user can cancel a QUEUED or PROVISIONING job, but not a
   393в†’  1392в†’  1392в†’RUNNING job (they must wait for it to complete). Job lifecycle state
   394в†’  1393в†’  1393в†’transitions are all recorded in the audit log.

   397в†’  1396в†’  1396в†’
   398в†’  1397в†’  1397в†’6.4.1 Registry Design
   399в†’  1398в†’  1398в†’
   400в†’  1399в†’  1399в†’The registry is an organization-scoped collection of model artifacts
   401в†’  1400в†’  1400в†’that have been signed, scanned, and approved for deployment. Each
   402в†’  1401в†’  1401в†’organization can create one or more registries (e.g., 'production',
   403в†’  1402в†’  1402в†’'staging', 'development'). Each registry has a deployment policy defined
   404в†’  1403в†’  1403в†’in OPA (Open Policy Agent) Rego language that specifies the conditions
   405в†’  1404в†’  1404в†’under which a model version is allowed to be published to that registry.
   406в†’  1405в†’  1405в†’A deployment policy is a set of rules that are evaluated against the
   407в†’  1406в†’  1406в†’model's metadata, scan results, signature status, and SBOM findings.
   408в†’  1407в†’  1407в†’
   409в†’  1408в†’  1408в†’A typical production registry policy might require: the model must have
   410в†’  1409в†’  1409в†’a valid signature from an approved signer identity (e.g., any user with
   411в†’  1410в†’  1410в†’the 'developer' role or higher), the sandbox scan must have completed
   412в†’  1411в†’  1411в†’with no CRITICAL findings and no more than 2 HIGH findings, the SBOM
   413в†’  1412в†’  1412в†’must have zero CRITICAL or HIGH CVEs, the model must have been uploaded
   414в†’  1413в†’  1413в†’by a member who has been in the organization for at least 7 days (to
   415в†’  1414в†’  1414в†’prevent insider threat from newly invited accounts), and the model
   416в†’  1415в†’  1415в†’format must be one of the approved formats (safetensors, onnx). If any
   417в†’  1416в†’  1416в†’of these conditions is not met, the deployment gate returns 'BLOCKED'
   418в†’  1417в†’  1417в†’with a detailed explanation of which policy rule failed and why.
   419в†’  1418в†’  1418в†’
   420в†’  1419в†’  1419в†’6.4.2 Policy Engine
   421в†’  1420в†’  1420в†’
   422в†’  1421в†’  1421в†’The policy engine is implemented using OPA (Open Policy Agent) embedded
   423в†’  1422в†’  1422в†’as a library within the registry-service. Each registry stores its
   424в†’  1423в†’  1423в†’policy as a Rego document in the database. When a user requests to
   425в†’  1424в†’  1424в†’publish a model or evaluate a deployment gate, the registry-service
   426в†’  1425в†’  1425в†’gathers all relevant data (model metadata, signature status, scan
   427в†’  1426в†’  1426в†’results, SBOM findings, user membership duration, organization settings)
   428в†’  1427в†’  1427в†’and constructs an input document. This input document is evaluated
   429в†’  1428в†’  1428в†’against the Rego policy using OPA's Eval function. The policy returns a
   430в†’  1429в†’  1429в†’decision (ALLOW, DENY) and a list of reasons. The decision, reasons, and
   431в†’  1430в†’  1430в†’full input document are stored in an immutable policy_evaluation record
   432в†’  1431в†’  1431в†’in the audit log.
   433в†’  1432в†’  1432в†’
   434в†’  1433в†’  1433в†’The OPA policy is exposed in the frontend as a code editor with Rego
   435в†’  1434в†’  1434в†’syntax highlighting and linting. Admins can write, test, and save
   436в†’  1435в†’  1435в†’policies directly in the UI. A 'Test Policy' button allows admins to
   437в†’  1436в†’  1436в†’evaluate a draft policy against a selected model version without
   438в†’  1437в†’  1437в†’publishing, showing the simulated decision and reasons. This allows
   439в†’  1438в†’  1438в†’policy authors to iterate on policies safely before they go live. When a
   440в†’  1439в†’  1439в†’policy is saved, the previous version is archived (never deleted) and
   441в†’  1440в†’  1440в†’the change is recorded in the audit log with the full before-and-after
   442в†’  1441в†’  1441в†’Rego source.

   449в†’  1448в†’  1448в†’(modelvault/security-scan) that integrates model verification into CI/CD
   450в†’  1449в†’  1449в†’pipelines. The action is distributed via the GitHub Marketplace and
   451в†’  1450в†’  1450в†’installed by adding a step to a workflow file. The action authenticates
   452в†’  1451в†’  1451в†’using an organization API key (stored as a GitHub secret), uploads the

   456в†’  1455в†’  1455в†’CRITICAL issues, the action fails the workflow, preventing the model

   462в†’  1461в†’  1461в†’results), 'scan-and-sign' (scan, and if it passes, automatically sign
   463в†’  1462в†’  1462в†’the model with the CI bot's identity), and 'scan-sign-publish' (scan,
   464в†’  1463в†’  1463в†’sign, and publish to a specified registry if the deployment gate
   465в†’  1464в†’  1464в†’passes). The mode is configured via the 'mode' action input parameter.
