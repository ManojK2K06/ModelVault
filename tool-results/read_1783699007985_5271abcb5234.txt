  1163→  1163→6. Feature Specifications
  1164→  1164→
  1165→  1165→6.1 Feature 1: Model Signing and Verification
  1166→  1166→
  1167→  1167→6.1.1 How Model Signing Works
  1168→  1168→
  1169→  1169→Model signing implements a Sigstore-inspired keyless signing workflow.
  1170→  1170→When a user clicks 'Sign Model' on a model version, the following
  1171→  1171→process executes: The signing-service receives a sign request containing
  1172→  1172→the model_version_id and the user's JWT access token. The service first
  1173→  1173→retrieves the model's SHA-256 and Merkle root hash from the
  1174→  1174→model_versions table. It then constructs a signing payload that includes
  1175→  1175→the model ID, version string, SHA-256 hash, Merkle root, file size,
  1176→  1176→format, and the current timestamp. This payload is serialized to JSON
  1177→  1177→and canonicalized (sorted keys, no whitespace) to ensure deterministic
  1178→  1178→hashing. The payload is then hashed with SHA-256 to produce the digest
  1179→  1179→that will be signed.
  1180→  1180→
  1181→  1181→The signing-service generates a short-lived ephemeral key pair (ECDSA
  1182→  1182→P-256) for this signing operation. It then contacts the Fulcio
  1183→  1183→certificate authority (or an internal Fulcio-compatible CA for
  1184→  1184→self-hosted deployments) with an OIDC identity token (obtained from the
  1185→  1185→user's session) and the ephemeral public key. Fulcio issues a
  1186→  1186→short-lived X.509 certificate that binds the ephemeral public key to the
  1187→  1187→user's OIDC identity (email). The signing-service signs the payload
  1188→  1188→digest with the ephemeral private key, producing an ECDSA-SHA256
  1189→  1189→signature. The signature, the X.509 certificate chain, and the signing
  1190→  1190→payload are all stored in the signatures table. The ephemeral private
  1191→  1191→key is then securely erased from memory. The certificate chain is also
  1192→  1192→submitted to a Rekor transparency log (or an internal immutable log for
  1193→  1193→self-hosted deployments), which returns a signed inclusion proof.
  1194→  1194→
  1195→  1195→Verification reverses this process: given a model file and a signature
  1196→  1196→record, the system recomputes the SHA-256 hash of the file, reconstructs
  1197→  1197→the signing payload, verifies the ECDSA signature against the X.509
  1198→  1198→certificate from the stored certificate chain, validates that the
  1199→  1199→certificate was issued by a trusted CA, checks that the certificate has
  1200→  1200→not expired, and optionally verifies the Rekor inclusion proof. If all
  1201→  1201→checks pass, the verification returns 'VERIFIED' with the signer's
  1202→  1202→identity. If any check fails, it returns 'FAILED' with a specific reason
  1203→  1203→(hash mismatch, signature invalid, certificate expired, untrusted CA,
  1204→  1204→etc.).
  1205→  1205→
  1206→  1206→6.1.2 Merkle Tree Integrity
  1207→  1207→
  1208→  1208→During upload, the model file is split into 4MB chunks. Each chunk is
  1209→  1209→hashed with SHA-256, producing a list of leaf hashes. These leaf hashes
  1210→  1210→are arranged into a binary Merkle tree: pairs of leaf hashes are
  1211→  1211→concatenated and hashed to produce parent nodes, up to a single root
  1212→  1212→hash. The Merkle root is stored in the model_versions table alongside
  1213→  1213→the flat SHA-256 of the complete file. During verification, both the
  1214→  1214→flat hash and the Merkle root are verified. The Merkle tree enables
  1215→  1215→efficient proof that a specific chunk of the model file has not been
  1216→  1216→tampered with, without requiring the entire file to be re-downloaded.
  1217→  1217→This is particularly useful for partial model updates and for verifying
  1218→  1218→models in distributed storage systems where different chunks may reside
  1219→  1219→on different nodes.
  1220→  1220→
  1221→  1221→6.2 Feature 2: AI-SBOM Generation
  1222→  1222→
  1223→  1223→6.2.1 How SBOM Generation Works
  1224→  1224→
  1225→  1225→AI-SBOM generation is a fully deterministic, non-AI process that uses
  1226→  1226→static analysis to extract metadata from model artifact files. When a
  1227→  1227→model is uploaded (or when the user explicitly triggers SBOM
  1228→  1228→generation), the sbom-service retrieves the model file from MinIO and
  1229→  1229→analyzes it based on its detected format. The analysis extracts: model
  1230→  1230→architecture metadata (number of layers, parameter count, tensor
  1231→  1231→dimensions, dtype), the complete dependency tree of Python packages
  1232→  1232→required to load and run the model (extracted from any embedded
  1233→  1233→requirements.txt, setup.py, pyproject.toml, or from the model card
  1234→  1234→metadata), a list of all third-party model components (base model
  1235→  1235→references, adapter configurations, tokenizer files), hardware
  1236→  1236→requirements (inferred from tensor sizes and dtype), and training data
  1237→  1237→provenance metadata if available from the model card or embedded
  1238→  1238→metadata.
  1239→  1239→
  1240→  1240→For SafeTensors format files, the parser reads the JSON header at the
  1241→  1241→start of the file, which contains the tensor metadata including names,
  1242→  1242→shapes, and dtypes. It calculates the total parameter count by
  1243→  1243→multiplying the dimensions of each tensor. It identifies the model
  1244→  1244→architecture by matching tensor naming patterns (e.g.,
  1245→  1245→'transformer.h.{N}.attn.c_attn.weight' indicates a GPT-2 style
  1246→  1246→architecture). For GGUF files, the parser reads the GGUF header which
  1247→  1247→contains key-value metadata including the model architecture,
  1248→  1248→quantization level, and embedding dimensions. For ONNX files, the parser
  1249→  1249→uses the ONNX protobuf schema to read the model graph, extracting
  1250→  1250→operation types, input/output shapes, and opset version. For PyTorch
  1251→  1251→.bin or .pt files, the parser loads the file using PyTorch's safe_load
  1252→  1252→function (which uses zip-based inspection without executing any code)
  1253→  1253→and extracts the state dict keys, tensor shapes, and dtypes.
  1254→  1254→
  1255→  1255→After extraction, the sbom-service cross-references all discovered
  1256→  1256→dependencies against the OSV (Open Source Vulnerabilities) database API
  1257→  1257→to identify known vulnerabilities. Each dependency is looked up by its
  1258→  1258→package name and version. If a CVE is found, it is recorded with its
  1259→  1259→severity (CRITICAL, HIGH, MEDIUM, LOW), CVSS score, and a link to the
  1260→  1260→advisory. The final SBOM is output in CycloneDX ML-BOM 1.0 JSON format,
  1261→  1261→which extends the standard CycloneDX format with ML-specific fields
  1262→  1262→including model architecture, parameter count, training data provenance,
  1263→  1263→and hardware requirements. The SBOM JSON is stored in the sboms table
  1264→  1264→and is also available for download via the API endpoint GET
  1265→  1265→/api/v1/models/{model_id}/sbom.
  1266→  1266→
  1267→  1267→6.3 Feature 3: Pre-Deployment Quarantine Sandbox
  1268→  1268→
  1269→  1269→6.3.1 Sandbox Architecture
  1270→  1270→
  1271→  1271→The quarantine sandbox is the most critical security feature of
  1272→  1272→ModelVault. It provides a completely isolated execution environment
  1273→  1273→where a model is loaded and exercised with a battery of behavioral tests
  1274→  1274→to detect malicious behavior before the model is approved for
  1275→  1275→deployment. The sandbox uses Firecracker microVMs, which are lightweight
  1276→  1276→virtual machines that boot in under 125 milliseconds with a dedicated
  1277→  1277→kernel, providing hardware-level isolation from the host system. Each
  1278→  1278→sandbox job gets its own Firecracker VM with a minimal root filesystem
  1279→  1279→(based on Alpine Linux, approximately 5MB), a Python runtime (CPython
  1280→  1280→3.12), a PyTorch/CUDA runtime (only CPU mode for security), and the
  1281→  1281→model file under test.
  1282→  1282→
  1283→  1283→The VM is configured with a strict network policy enforced by iptables:
  1284→  1284→all outbound network connections are blocked by default. The only
  1285→  1285→allowed outbound traffic is a DNS resolution to a mock DNS server that
  1286→  1286→returns NXDOMAIN for all queries, used to test whether the model
  1287→  1287→attempts to perform DNS lookups (which would indicate potential data
  1288→  1288→exfiltration). All file system access is monitored using eBPF programs
  1289→  1289→attached to the VFS layer, which capture every open(), read(), write(),
  1290→  1290→and unlink() syscall with the full path, flags, and return value. The
  1291→  1291→eBPF programs are compiled from BPF C code using the bcc toolkit and
  1292→  1292→loaded into the VM's kernel at boot time. Syscall events are streamed to
  1293→  1293→a sidecar process over a virtio-serial channel, which aggregates them
  1294→  1294→and writes the structured event stream to the sandbox_jobs.result_json
  1295→  1295→field in the database.
  1296→  1296→

  1378→  1378→6.3.3 Sandbox Job Lifecycle
  1379→  1379→
  1380→  1380→A sandbox job goes through the following lifecycle states: QUEUED (job
  1381→  1381→created, waiting for a VM slot), PROVISIONING (Firecracker VM is being
  1382→  1382→created and the model file is being copied into the VM's rootfs),
  1383→  1383→RUNNING (all behavioral tests are executing inside the VM), COMPLETED
  1384→  1384→(all tests finished, results captured, VM terminated), FAILED (VM
  1385→  1385→crashed, timeout, or internal error), CANCELLED (user cancelled the job
  1386→  1386→before completion). The sandbox-service maintains a pool of
  1387→  1387→pre-provisioned VM slots (configurable, default 4 per node) to minimize
  1388→  1388→provisioning latency. When a job is QUEUED, it is assigned to the next
  1389→  1389→available slot. If no slot is available within 30 seconds, the job
  1390→  1390→remains in QUEUED status and the frontend displays a queue position
  1391→  1391→indicator. The user can cancel a QUEUED or PROVISIONING job, but not a
  1392→  1392→RUNNING job (they must wait for it to complete). Job lifecycle state
  1393→  1393→transitions are all recorded in the audit log.
  1394→  1394→
  1395→  1395→6.4 Feature 4: Secure Model Artifact Registry
  1396→  1396→
  1397→  1397→6.4.1 Registry Design
  1398→  1398→
  1399→  1399→The registry is an organization-scoped collection of model artifacts
  1400→  1400→that have been signed, scanned, and approved for deployment. Each
  1401→  1401→organization can create one or more registries (e.g., 'production',
  1402→  1402→'staging', 'development'). Each registry has a deployment policy defined
  1403→  1403→in OPA (Open Policy Agent) Rego language that specifies the conditions
  1404→  1404→under which a model version is allowed to be published to that registry.
  1405→  1405→A deployment policy is a set of rules that are evaluated against the
  1406→  1406→model's metadata, scan results, signature status, and SBOM findings.
  1407→  1407→
  1408→  1408→A typical production registry policy might require: the model must have
  1409→  1409→a valid signature from an approved signer identity (e.g., any user with
  1410→  1410→the 'developer' role or higher), the sandbox scan must have completed

  1419→  1419→6.4.2 Policy Engine
  1420→  1420→
  1421→  1421→The policy engine is implemented using OPA (Open Policy Agent) embedded
  1422→  1422→as a library within the registry-service. Each registry stores its
  1423→  1423→policy as a Rego document in the database. When a user requests to
  1424→  1424→publish a model or evaluate a deployment gate, the registry-service
  1425→  1425→gathers all relevant data (model metadata, signature status, scan
  1426→  1426→results, SBOM findings, user membership duration, organization settings)
  1427→  1427→and constructs an input document. This input document is evaluated
  1428→  1428→against the Rego policy using OPA's Eval function. The policy returns a
  1429→  1429→decision (ALLOW, DENY) and a list of reasons. The decision, reasons, and
  1430→  1430→full input document are stored in an immutable policy_evaluation record
  1431→  1431→in the audit log.
  1432→  1432→
  1433→  1433→The OPA policy is exposed in the frontend as a code editor with Rego
  1434→  1434→syntax highlighting and linting. Admins can write, test, and save
  1435→  1435→policies directly in the UI. A 'Test Policy' button allows admins to
  1436→  1436→evaluate a draft policy against a selected model version without
  1437→  1437→publishing, showing the simulated decision and reasons. This allows
  1438→  1438→policy authors to iterate on policies safely before they go live. When a
  1439→  1439→policy is saved, the previous version is archived (never deleted) and
  1440→  1440→the change is recorded in the audit log with the full before-and-after
  1441→  1441→Rego source.

  1446→  1446→
  1447→  1447→ModelVault provides a GitHub Actions composite action
  1448→  1448→(modelvault/security-scan) that integrates model verification into CI/CD
  1449→  1449→pipelines. The action is distributed via the GitHub Marketplace and

  1451→  1451→using an organization API key (stored as a GitHub secret), uploads the
  1452→  1452→model file to ModelVault, triggers a full scan (SBOM generation +
  1453→  1453→sandbox behavioral tests), waits for the scan to complete (with
  1454→  1454→configurable timeout), and checks the scan result. If the scan finds any
  1455→  1455→CRITICAL issues, the action fails the workflow, preventing the model
  1456→  1456→from being deployed. If the scan passes, the action returns the model
  1457→  1457→ID, version ID, and signature URL as outputs that downstream workflow
  1458→  1458→steps can reference.
  1459→  1459→
  1460→  1460→The action supports three modes: 'scan-only' (upload, scan, report
  1461→  1461→results), 'scan-and-sign' (scan, and if it passes, automatically sign
  1462→  1462→the model with the CI bot's identity), and 'scan-sign-publish' (scan,
  1463→  1463→sign, and publish to a specified registry if the deployment gate
  1464→  1464→passes). The mode is configured via the 'mode' action input parameter.
  1465→  1465→The action also supports uploading models from specific paths, from
