     1в†’   500в†’   500в†’
     2в†’   501в†’   501в†’All application secrets, database credentials, API keys, TLS
     3в†’   502в†’   502в†’certificates, and signing key material are stored in HashiCorp Vault
     4в†’   503в†’   503в†’with the Transit secrets engine for encryption-as-a-service. Application
     5в†’   504в†’   504в†’services access secrets through the Vault Agent sidecar pattern: a Vault
     6в†’   505в†’   505в†’Agent container runs alongside each application pod, authenticates to
     7в†’   506в†’   506в†’Vault using the Kubernetes auth method (each pod gets a unique JWT bound
     8в†’   507в†’   507в†’to its service account), and writes the current secret values to a
     9в†’   508в†’   508в†’shared memory-backed volume. Secrets are never written to disk. The
    10в†’   509в†’   509в†’Vault Agent automatically renews leases and re-fetches secrets before
    11в†’   510в†’   510в†’they expire. In production deployments, the Vault unseal key shares are
    12в†’   511в†’   511в†’distributed across multiple HSM appliances using Shamir's Secret Sharing
    13в†’   512в†’   512в†’with a 5-of-7 threshold, ensuring no single person or machine can unseal
    14в†’   513в†’   513в†’Vault independently.
    15в†’   514в†’   514в†’
    16в†’   515в†’   515в†’3.4 Application Security
    17в†’   516в†’   516в†’
    18в†’   517в†’   517в†’3.4.1 Input Validation and Sanitization
    19в†’   518в†’   518в†’
    20в†’   519в†’   519в†’Every API endpoint validates its input against a strict JSON Schema
    21в†’   520в†’   520в†’definition before processing. The JSON Schema validation runs in the
    22в†’   521в†’   521в†’api-gateway as a WASM filter, ensuring that malformed, oversized, or
    23в†’   522в†’   522в†’malicious payloads are rejected before reaching any backend service.
    24в†’   523в†’   523в†’String fields are validated for length (min/max), character set
    25в†’   524в†’   524в†’(allowlisting, not denylisting), and format (email, URL, UUID, semver).
    26в†’   525в†’   525в†’File upload endpoints validate the file's magic bytes against the
    27в†’   526в†’   526в†’expected format (e.g., SafeTensors files must begin with the JSON header
    28в†’   527в†’   527в†’followed by the safetensors magic bytes 0x6C6F7261). Uploaded files are
    29в†’   528в†’   528в†’stored in a staging area with a random UUID filename (never the original
    30в†’   529в†’   529в†’filename) and are not processed until the upload is complete and the
    31в†’   530в†’   530в†’checksum matches. All user-supplied text is sanitized to prevent stored
    32в†’   531в†’   531в†’XSS: HTML tags are stripped, and special characters are entity-encoded
    33в†’   532в†’   532в†’before storage.
    34в†’   533в†’   533в†’
    35в†’   534в†’   534в†’3.4.2 SQL Injection Prevention
    36в†’   535в†’   535в†’
    37в†’   536в†’   536в†’All database queries use parameterized prepared statements via pgx (the
    38в†’   537в†’   537в†’Go PostgreSQL driver). Raw string interpolation in SQL queries is
    39в†’   538в†’   538в†’prohibited by code review policy and enforced by a custom linter rule
    40в†’   539в†’   539в†’that scans for fmt.Sprintf or string concatenation in database query
    41в†’   540в†’   540в†’contexts. Additionally, the PostgreSQL user account used by each
    42в†’   541в†’   541в†’microservice is granted only the minimum required privileges: SELECT,
    43в†’   542в†’   542в†’INSERT, UPDATE on specific tables, with no DDL permissions. DDL
    44в†’   543в†’   543в†’migrations are executed only through a dedicated migration service that
    45в†’   544в†’   544в†’runs with elevated privileges during deployment and has no runtime API
    46в†’   545в†’   545в†’access.
    47в†’   546в†’   546в†’
    48в†’   547в†’   547в†’3.4.3 Cross-Site Request Forgery (CSRF) Protection
    49в†’   548в†’   548в†’
    50в†’   549в†’   549в†’The frontend uses SameSite=Strict cookies for the refresh token, which
    51в†’   550в†’   550в†’prevents CSRF attacks because the browser will not send the cookie with
    52в†’   551в†’   551в†’cross-site requests. Additionally, the api-gateway checks the Origin and
    53в†’   552в†’   552в†’Referer headers on all state-changing requests (POST, PUT, DELETE,
    54в†’   553в†’   553в†’PATCH) and rejects requests where the origin does not match the expected
    55в†’   554в†’   554в†’frontend domain. For API-only clients (CLI tools, CI/CD integrations),
    56в†’   555в†’   555в†’CSRF protection is handled via the OAuth 2.0 client credentials flow,
    57в†’   556в†’   556в†’which uses bearer tokens instead of cookies.
    58в†’   557в†’   557в†’
    59в†’   558в†’   558в†’3.4.4 Dependency Security
    60в†’   559в†’   559в†’
    61в†’   560в†’   560в†’All third-party dependencies are pinned to exact versions in go.mod and
    62в†’   561в†’   561в†’package-lock.json. A nightly CI job runs Trivy and Grype scans on the
    63в†’   562в†’   562в†’container images for all services, reporting any CVEs with severity
    64в†’   563в†’   563в†’CRITICAL or HIGH to the security team. Dependencies with known CRITICAL
    65в†’   564в†’   564в†’CVEs are blocked from deployment via a CI/CD policy gate. The base
    66в†’   565в†’   565в†’container images use distroless variants
    67в†’   566в†’   566в†’(gcr.io/distroless/static-debian12) that contain no shell, no package
    68в†’   567в†’   567в†’manager, and no unnecessary binaries, reducing the attack surface of the
    69в†’   568в†’   568в†’container runtime.
    70в†’   569в†’   569в†’
    71в†’   570в†’   570в†’3.5 Audit Logging
    72в†’   571в†’   571в†’
    73в†’   572в†’   572в†’Every action performed in ModelVault is recorded in an immutable audit
    74в†’   573в†’   573в†’log stored in a separate PostgreSQL database (audit_db) with the
    75в†’   574в†’   574в†’following properties: append-only table (no UPDATE or DELETE permissions
    76в†’   575в†’   575в†’granted to any service account), each log entry is cryptographically
    77в†’   576в†’   576в†’chained to the previous entry using a hash of the previous record's
    78в†’   577в†’   577в†’content plus a server-side timestamp, making it tamper-evident. The
    79в†’   578в†’   578в†’audit log captures: the actor (user ID, service account ID, or system),
    80в†’   579в†’   579в†’the action (API method, internal event), the target resource (model ID,
    81в†’   580в†’   580в†’policy ID, user ID), the outcome (success or failure with error code),
    82в†’   581в†’   581в†’the source IP address, the user agent, and a full snapshot of the
    83в†’   582в†’   582в†’request payload (with sensitive fields like passwords and tokens
    84в†’   583в†’   583в†’redacted). Audit logs are retained for a minimum of 2 years and are
    85в†’   584в†’   584в†’replicated to a read replica in a separate availability zone for
    86в†’   585в†’   585в†’disaster recovery. A daily integrity verification job hashes all audit
    87в†’   586в†’   586в†’log entries from the previous day and compares the chain hash against
    88в†’   587в†’   587в†’the expected value, alerting the security team if any discrepancy is
    89в†’   588в†’   588в†’detected.

   292в†’   791в†’   791в†’4.6 Key File Upload Process (TUS Resumable)
   293в†’   792в†’   792в†’
   294в†’   793в†’   793в†’Model file uploads use the TUS protocol to handle files up to 50GB
   295в†’   794в†’   794в†’reliably. The process works as follows: First, the client sends a POST
   296в†’   795в†’   795в†’to /api/v1/models/upload with headers specifying the file name, total
   297в†’   796в†’   796в†’size (Content-Length), and model format (X-Model-Format:
   298в†’   797в†’   797в†’safetensors|gguf|onnx|pytorch|custom). The server validates that the
   299в†’   798в†’   798в†’organization has not exceeded its storage quota, creates a database
   300в†’   799в†’   799в†’record for the upload with status 'pending', and returns a Location
   301в†’   800в†’   800в†’header with the upload URL containing a UUID upload_id. The server also
   302в†’   801в†’   801в†’returns an Upload-Length header confirming the expected size.
   303в†’   802в†’   802в†’
   304в†’   803в†’   803в†’Second, the client uploads the file in chunks by sending PATCH requests
   305в†’   804в†’   804в†’to the upload URL. Each PATCH includes a Content-Range header (e.g.,
   306в†’   805в†’   805в†’'bytes 0-5242879/524288000') and the raw binary chunk as the request
   307в†’   806в†’   806в†’body. The server writes each chunk to a staging area in MinIO using the
   308в†’   807в†’   807в†’multipart upload API. Each chunk is checksummed with SHA-256, and the
   309в†’   808в†’   808в†’partial checksums are stored in Redis. If a chunk upload fails (network
   310в†’   809в†’   809в†’error, timeout), the client retries that specific chunk. TUS handles
   311в†’   810в†’   810в†’this via the Upload-Offset header: the server tracks how many bytes have
   312в†’   811в†’   811в†’been received and tells the client where to resume.
   313в†’   812в†’   812в†’
   314в†’   813в†’   813в†’Third, when the final chunk is uploaded, the server concatenates all
   315в†’   814в†’   814в†’parts in MinIO, computes the final SHA-256 checksum of the complete
   316в†’   815в†’   815в†’file, and compares it against the checksum provided by the client in the
   317в†’   816в†’   816в†’Upload-Metadata header. If the checksums match, the upload status is set
   318в†’   817в†’   817в†’to 'complete', the file is moved from the staging bucket to the
   319в†’   818в†’   818в†’permanent storage bucket with a UUID-based path (e.g.,
   320в†’   819в†’   819в†’org/{org_id}/models/{model_id}/v1/artifact), and a model_version record
   321в†’   820в†’   820в†’is created in the database with the file size, format, SHA-256 hash,
   322в†’   821в†’   821в†’storage path, and upload timestamp. If the checksum does not match, the
   323в†’   822в†’   822в†’upload is marked 'failed' and all staged parts are deleted. The entire
   324в†’   823в†’   823в†’upload process generates audit log entries for initiation, each chunk,
   325в†’   824в†’   824в†’completion, or failure.

   329в†’   828в†’   828в†’5.1 Schema Overview
   330в†’   829в†’   829в†’
   331в†’   830в†’   830в†’The database uses PostgreSQL 16 with nine logical databases (one per
   332в†’   831в†’   831в†’microservice plus a shared auth database). Each microservice owns its
   333в†’   832в†’   832в†’schema and has no cross-schema query access. Data that must be shared
   334в†’   833в†’   833в†’between services is replicated asynchronously via Kafka events. All
   335в†’   834в†’   834в†’tables use UUIDv4 primary keys. All timestamps are stored as TIMESTAMPTZ
   336в†’   835в†’   835в†’in UTC. Row-level security (RLS) is enabled on all tables that contain
   337в†’   836в†’   836в†’organization-scoped data, ensuring that a compromised service account
   338в†’   837в†’   837в†’can only access data belonging to the organization it is authorized for.
