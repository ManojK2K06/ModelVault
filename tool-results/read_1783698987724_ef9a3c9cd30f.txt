   500тЖТ   500тЖТ
   501тЖТ   501тЖТAll application secrets, database credentials, API keys, TLS
   502тЖТ   502тЖТcertificates, and signing key material are stored in HashiCorp Vault
   503тЖТ   503тЖТwith the Transit secrets engine for encryption-as-a-service. Application
   504тЖТ   504тЖТservices access secrets through the Vault Agent sidecar pattern: a Vault
   505тЖТ   505тЖТAgent container runs alongside each application pod, authenticates to
   506тЖТ   506тЖТVault using the Kubernetes auth method (each pod gets a unique JWT bound
   507тЖТ   507тЖТto its service account), and writes the current secret values to a
   508тЖТ   508тЖТshared memory-backed volume. Secrets are never written to disk. The
   509тЖТ   509тЖТVault Agent automatically renews leases and re-fetches secrets before
   510тЖТ   510тЖТthey expire. In production deployments, the Vault unseal key shares are
   511тЖТ   511тЖТdistributed across multiple HSM appliances using Shamir's Secret Sharing
   512тЖТ   512тЖТwith a 5-of-7 threshold, ensuring no single person or machine can unseal
   513тЖТ   513тЖТVault independently.
   514тЖТ   514тЖТ
   515тЖТ   515тЖТ3.4 Application Security
   516тЖТ   516тЖТ
   517тЖТ   517тЖТ3.4.1 Input Validation and Sanitization
   518тЖТ   518тЖТ
   519тЖТ   519тЖТEvery API endpoint validates its input against a strict JSON Schema
   520тЖТ   520тЖТdefinition before processing. The JSON Schema validation runs in the
   521тЖТ   521тЖТapi-gateway as a WASM filter, ensuring that malformed, oversized, or
   522тЖТ   522тЖТmalicious payloads are rejected before reaching any backend service.
   523тЖТ   523тЖТString fields are validated for length (min/max), character set
   524тЖТ   524тЖТ(allowlisting, not denylisting), and format (email, URL, UUID, semver).
   525тЖТ   525тЖТFile upload endpoints validate the file's magic bytes against the
   526тЖТ   526тЖТexpected format (e.g., SafeTensors files must begin with the JSON header
   527тЖТ   527тЖТfollowed by the safetensors magic bytes 0x6C6F7261). Uploaded files are
   528тЖТ   528тЖТstored in a staging area with a random UUID filename (never the original
   529тЖТ   529тЖТfilename) and are not processed until the upload is complete and the
   530тЖТ   530тЖТchecksum matches. All user-supplied text is sanitized to prevent stored
   531тЖТ   531тЖТXSS: HTML tags are stripped, and special characters are entity-encoded
   532тЖТ   532тЖТbefore storage.
   533тЖТ   533тЖТ
   534тЖТ   534тЖТ3.4.2 SQL Injection Prevention
   535тЖТ   535тЖТ
   536тЖТ   536тЖТAll database queries use parameterized prepared statements via pgx (the
   537тЖТ   537тЖТGo PostgreSQL driver). Raw string interpolation in SQL queries is
   538тЖТ   538тЖТprohibited by code review policy and enforced by a custom linter rule
   539тЖТ   539тЖТthat scans for fmt.Sprintf or string concatenation in database query
   540тЖТ   540тЖТcontexts. Additionally, the PostgreSQL user account used by each
   541тЖТ   541тЖТmicroservice is granted only the minimum required privileges: SELECT,
   542тЖТ   542тЖТINSERT, UPDATE on specific tables, with no DDL permissions. DDL
   543тЖТ   543тЖТmigrations are executed only through a dedicated migration service that
   544тЖТ   544тЖТruns with elevated privileges during deployment and has no runtime API
   545тЖТ   545тЖТaccess.
   546тЖТ   546тЖТ
   547тЖТ   547тЖТ3.4.3 Cross-Site Request Forgery (CSRF) Protection
   548тЖТ   548тЖТ
   549тЖТ   549тЖТThe frontend uses SameSite=Strict cookies for the refresh token, which
   550тЖТ   550тЖТprevents CSRF attacks because the browser will not send the cookie with
   551тЖТ   551тЖТcross-site requests. Additionally, the api-gateway checks the Origin and
   552тЖТ   552тЖТReferer headers on all state-changing requests (POST, PUT, DELETE,
   553тЖТ   553тЖТPATCH) and rejects requests where the origin does not match the expected
   554тЖТ   554тЖТfrontend domain. For API-only clients (CLI tools, CI/CD integrations),
   555тЖТ   555тЖТCSRF protection is handled via the OAuth 2.0 client credentials flow,
   556тЖТ   556тЖТwhich uses bearer tokens instead of cookies.
   557тЖТ   557тЖТ
   558тЖТ   558тЖТ3.4.4 Dependency Security
   559тЖТ   559тЖТ
   560тЖТ   560тЖТAll third-party dependencies are pinned to exact versions in go.mod and
   561тЖТ   561тЖТpackage-lock.json. A nightly CI job runs Trivy and Grype scans on the
   562тЖТ   562тЖТcontainer images for all services, reporting any CVEs with severity
   563тЖТ   563тЖТCRITICAL or HIGH to the security team. Dependencies with known CRITICAL
   564тЖТ   564тЖТCVEs are blocked from deployment via a CI/CD policy gate. The base
   565тЖТ   565тЖТcontainer images use distroless variants
   566тЖТ   566тЖТ(gcr.io/distroless/static-debian12) that contain no shell, no package
   567тЖТ   567тЖТmanager, and no unnecessary binaries, reducing the attack surface of the
   568тЖТ   568тЖТcontainer runtime.
   569тЖТ   569тЖТ
   570тЖТ   570тЖТ3.5 Audit Logging
   571тЖТ   571тЖТ
   572тЖТ   572тЖТEvery action performed in ModelVault is recorded in an immutable audit
   573тЖТ   573тЖТlog stored in a separate PostgreSQL database (audit_db) with the
   574тЖТ   574тЖТfollowing properties: append-only table (no UPDATE or DELETE permissions
   575тЖТ   575тЖТgranted to any service account), each log entry is cryptographically
   576тЖТ   576тЖТchained to the previous entry using a hash of the previous record's
   577тЖТ   577тЖТcontent plus a server-side timestamp, making it tamper-evident. The
   578тЖТ   578тЖТaudit log captures: the actor (user ID, service account ID, or system),
   579тЖТ   579тЖТthe action (API method, internal event), the target resource (model ID,
   580тЖТ   580тЖТpolicy ID, user ID), the outcome (success or failure with error code),
   581тЖТ   581тЖТthe source IP address, the user agent, and a full snapshot of the
   582тЖТ   582тЖТrequest payload (with sensitive fields like passwords and tokens
   583тЖТ   583тЖТredacted). Audit logs are retained for a minimum of 2 years and are
   584тЖТ   584тЖТreplicated to a read replica in a separate availability zone for
   585тЖТ   585тЖТdisaster recovery. A daily integrity verification job hashes all audit
   586тЖТ   586тЖТlog entries from the previous day and compares the chain hash against
   587тЖТ   587тЖТthe expected value, alerting the security team if any discrepancy is
   588тЖТ   588тЖТdetected.
   589тЖТ   589тЖТ
   590тЖТ   590тЖТ4. Backend API Specification
   591тЖТ   591тЖТ
   592тЖТ   592тЖТ4.1 API Design Principles
   593тЖТ   593тЖТ
   594тЖТ   594тЖТAll external APIs follow RESTful conventions with OpenAPI 3.1
   595тЖТ   595тЖТspecifications. Internal service-to-service communication uses gRPC with
   596тЖТ   596тЖТProtocol Buffers. The API versioning strategy uses URL path versioning
   597тЖТ   597тЖТ(e.g., /api/v1/models). All responses use consistent JSON envelopes with
   598тЖТ   598тЖТthe structure: { status, data, meta, errors }. Pagination uses
   599тЖТ   599тЖТcursor-based pagination (not offset-based) for all list endpoints,
   600тЖТ   600тЖТreturning a next_cursor field. All timestamps are in UTC ISO 8601 format
   601тЖТ   601тЖТwith millisecond precision. All IDs are UUIDv4. File upload endpoints
   602тЖТ   602тЖТuse resumable uploads via the TUS protocol (tus.io) to support large
   603тЖТ   603тЖТmodel files up to 50GB with automatic retry on network interruption.

   791тЖТ   791тЖТ4.6 Key File Upload Process (TUS Resumable)
   792тЖТ   792тЖТ
   793тЖТ   793тЖТModel file uploads use the TUS protocol to handle files up to 50GB
   794тЖТ   794тЖТreliably. The process works as follows: First, the client sends a POST
   795тЖТ   795тЖТto /api/v1/models/upload with headers specifying the file name, total
   796тЖТ   796тЖТsize (Content-Length), and model format (X-Model-Format:
   797тЖТ   797тЖТsafetensors|gguf|onnx|pytorch|custom). The server validates that the
   798тЖТ   798тЖТorganization has not exceeded its storage quota, creates a database
   799тЖТ   799тЖТrecord for the upload with status 'pending', and returns a Location
   800тЖТ   800тЖТheader with the upload URL containing a UUID upload_id. The server also
   801тЖТ   801тЖТreturns an Upload-Length header confirming the expected size.
   802тЖТ   802тЖТ
   803тЖТ   803тЖТSecond, the client uploads the file in chunks by sending PATCH requests
   804тЖТ   804тЖТto the upload URL. Each PATCH includes a Content-Range header (e.g.,
   805тЖТ   805тЖТ'bytes 0-5242879/524288000') and the raw binary chunk as the request
   806тЖТ   806тЖТbody. The server writes each chunk to a staging area in MinIO using the
   807тЖТ   807тЖТmultipart upload API. Each chunk is checksummed with SHA-256, and the
   808тЖТ   808тЖТpartial checksums are stored in Redis. If a chunk upload fails (network
   809тЖТ   809тЖТerror, timeout), the client retries that specific chunk. TUS handles
   810тЖТ   810тЖТthis via the Upload-Offset header: the server tracks how many bytes have
   811тЖТ   811тЖТbeen received and tells the client where to resume.
   812тЖТ   812тЖТ
   813тЖТ   813тЖТThird, when the final chunk is uploaded, the server concatenates all
   814тЖТ   814тЖТparts in MinIO, computes the final SHA-256 checksum of the complete
   815тЖТ   815тЖТfile, and compares it against the checksum provided by the client in the
   816тЖТ   816тЖТUpload-Metadata header. If the checksums match, the upload status is set
   817тЖТ   817тЖТto 'complete', the file is moved from the staging bucket to the
   818тЖТ   818тЖТpermanent storage bucket with a UUID-based path (e.g.,
   819тЖТ   819тЖТorg/{org_id}/models/{model_id}/v1/artifact), and a model_version record
   820тЖТ   820тЖТis created in the database with the file size, format, SHA-256 hash,
   821тЖТ   821тЖТstorage path, and upload timestamp. If the checksum does not match, the
   822тЖТ   822тЖТupload is marked 'failed' and all staged parts are deleted. The entire
   823тЖТ   823тЖТupload process generates audit log entries for initiation, each chunk,
   824тЖТ   824тЖТcompletion, or failure.
   825тЖТ   825тЖТ
   826тЖТ   826тЖТ5. Database Schema
   827тЖТ   827тЖТ
   828тЖТ   828тЖТ5.1 Schema Overview
   829тЖТ   829тЖТ
   830тЖТ   830тЖТThe database uses PostgreSQL 16 with nine logical databases (one per
   831тЖТ   831тЖТmicroservice plus a shared auth database). Each microservice owns its
   832тЖТ   832тЖТschema and has no cross-schema query access. Data that must be shared
